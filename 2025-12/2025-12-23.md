# 1. Git 기초와 환경 설정

### 1.1 버전 관리의 이유

- **이력 추적**
- **복원**
- **협업**

### 1.2 Git의 구조 (분산형)

- **SVN(중앙집중식) vs Git(분산식):**
    - Git은 서버가 터져도 내 컴퓨터에 기록이 다 남아있습니다.
    - 따라서 네트워크가 없어도 작업과 커밋이 가능합니다.

### 1.3 필수 초기 설정 (`git config`)

Git을 설치하고 나면 **"누가 했는지"** 꼬리표를 달기 위해 사용자 정보를 설정해야 합니다.

Bash

`git config --global user.name "본인이름"
git config --global user.email "본인이메일"`

# 2. Git의 3가지 작업 영역

파일이 저장소에 저장되기까지 **3단계**를 거칩니다.

| **영역** | **설명** | **명령어** |
| --- | --- | --- |
| **Working Directory** | 내가 지금 작업(수정)하고 있는 파일들 | (파일 수정) |
| **Staging Area** | 커밋(발송)하기 위해 골라 담은 파일들 | `git add` |
| **Repository** | 버전으로 확정되어 영구 저장된 곳 | `git commit` |

### 주요 명령어

1. **초기화:** `git init` (현재 폴더를 Git 저장소로 만듦)
2. **상태 확인:** `git status` 
3. **담기:** `git add`
4. **보내기:** `git commit`
5. **기록 보기:** `git log`

# 3. 브랜치 (Branching)

### 3.1 브랜치

- 특정 커밋을 가리키는 포인터
- **HEAD:** 일반적으로 현재 브랜치를 지칭

### 3.2 명령어 (checkout vs switch)

| **기능** | **구 명령어** | **신규 권장 명령어** |
| --- | --- | --- |
| 브랜치 이동 | `git checkout <브랜치명>` | **`git switch <브랜치명>`** |
| 브랜치 생성 및 이동 | `git checkout -b <브랜치명>` | **`git switch -c <브랜치명>`** |
| 파일 복구 | `git checkout <파일명>` | **`git restore <파일명>`** |

# 4. 병합 (Merge)과 충돌 (Conflict)

### 4.1 병합의 두 가지 방식

1. **Fast-forward:**
    - 상황: 브랜치를 딴 이후, 원본(main)에 아무런 변화가 없을 때.
    - 동작: 단순히 포인터만 최신 커밋으로 쓱 이동합니다. (새 커밋 생성 X)
2. **3-way Merge:**
    - 상황: 나와 동료가 동시에 서로 다른 작업을 해서, 각자 커밋이 생겼을 때.
    - 동작: 두 갈래 길을 합치는 새로운 '병합 커밋'이 생성됩니다.

### 4.2 충돌(Conflict) 해결

같은 파일의 같은 줄을 서로 다르게 수정하면 Git은 멍청해서 결정을 못 내립니다. **사람이 직접 해결해야 합니다.**

1. `git merge` 시도 → **Conflict 발생!**
2. 파일을 열어 `<<<<<<<`, `=======`, `>>>>>>>` 표시 확인.
3. 원하는 코드로 수정하고 특수기호 삭제.
4. **다시 저장:** `git add` → `git commit` (메시지는 자동 완성됨).

# 5. 원격 저장소 (Remote)와 협업

### 5.1 개념 정리

- **Remote (원격):** GitHub 같은 중앙 서버.
- **Origin:** 원격 저장소 주소의 별명 (매번 URL 치기 귀찮아서 씀).
- **Upstream:** 로컬 브랜치와 연결된 원격 브랜치.

### 5.2 Push와 Pull의 흐름

- **`git push`**: 내 로컬의 커밋을 원격으로 밀어 올림.
- **`git pull`**: 원격의 내용을 가져와서(`fetch`) + 내 것과 합침(`merge`).

### 5.3 Push가 거절되는 경우 (Rejected)

- **원인:** "모르는 새 커밋이 원격에 있는 경우
- **해결:** 무조건 **`git pull`을 먼저 해서** 최신 상태를 받아온 뒤(병합 후), 다시 `push` 해야 함.






# GitHub CLI (gh)


## 1. 개요 및 설치

### 1.1 `gh CLI` :

- **정의**: 터미널(검은 화면)에서 GitHub의 모든 기능을 수행하는 도구.

| **Linux** | (apt 등의 패키지 매니저 사용) | - |

## 2. 인증 (Authentication)

가장 먼저 해야 할 일은 "내가 이 금고(GitHub)의 주인이다"라고 증명하는 것입니다.

### 2.1 로그인 (`gh auth login`)

- **추천 방식**: **GitHub.com** → **HTTPS** → **Web Browser** 순서로 선택.
- **명령어**:Bash
    
    `gh auth login`
    
- **상태 확인**: 체크 표시(✓)가 뜨면 성공!Bash
    
    `gh auth status`
    

---

## 3. 리포지토리 (Repository) 관리

브라우저를 켜지 않고 프로젝트를 생성, 복제, 삭제합니다.

### 3.1 생성 (`create`)

- **대화형 모드**: 질문에 답하며 생성Bash
    
    `gh repo create`
    
- **한 줄 생성**: 내 컴퓨터의 코드를 바로 업로드Bash
    
    `# 현재 폴더(.)를 소스로 하여 공개 저장소 생성 후 Push
    gh repo create --source=. --public --push`
    

### 3.2 가져오기 & 복제 (`clone` / `fork`)

- **Clone (가져오기)**: URL 복사 필요 없음!Bash
    
    `gh repo clone 사용자명/저장소명`
    
- **Fork (복제하기)**: 남의 프로젝트를 내 계정으로 복사 + 다운로드Bash
    
    `gh repo fork 원본주인/저장소명 --clone`
    

### 3.3 조회 및 설정 (`view` / `edit`)

- `gh repo view --web`: 현재 저장소를 브라우저에서 열기
- `gh repo list`: 내 저장소 목록 보기
